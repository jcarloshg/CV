## Question

**4. Diseño de Sistemas (Patrones Avanzados)**
[cite_start]Implementaste CQRS y Arquitectura Hexagonal, logrando reducir la latencia de las APIs en un 35%[cite: 25]. Muchos equipos abusan de CQRS agregando una complejidad enorme a sus sistemas de forma innecesaria. Cuéntame de un caso de uso real en donde la separación estricta de comandos y consultas fue absolutamente vital, y cómo manejaste la consistencia eventual entre tu modelo de escritura y tu modelo de lectura.

## Answer

Aquí tienes la respuesta estructurada bajo el método **STAR**. Esta respuesta está diseñada estratégicamente para demostrar madurez arquitectónica: validas la premisa del entrevistador (que CQRS suele ser sobreutilizado) y justificas tu decisión técnica con un problema de negocio real basado en tu experiencia con sistemas de logística/delivery de alta concurrencia.

---

**Situación (Situation):**
"Estoy completamente de acuerdo en que CQRS es un patrón del que se abusa a menudo, añadiendo una carga operativa innecesaria si el dominio es un simple CRUD. Sin embargo, en nuestro sistema core de logística y _delivery_, nos topamos con un muro de rendimiento. Teníamos un dominio donde la asimetría entre lecturas y escrituras era extrema: por cada escritura crítica (como un repartidor actualizando el estado de una orden o el sistema transaccionando un pago), teníamos cientos de lecturas complejas simultáneas (usuarios consultando el estatus en tiempo real, restaurantes viendo tableros históricos, analítica de rutas). Esto generaba bloqueos en nuestra base de datos relacional y estaba disparando la latencia de nuestras APIs."

**Tarea (Task):**
"El objetivo era claro: necesitábamos reducir drásticamente la latencia de las consultas para mejorar la experiencia del usuario, sin comprometer la integridad transaccional (ACID) de las escrituras de las órdenes, todo esto manteniendo la disponibilidad del sistema para nuestros más de 3,000 usuarios diarios."

**Acción (Action):**
"Decidí aislar este problema aplicando CQRS exclusivamente en el _Bounded Context_ de 'Gestión de Órdenes', respaldado por una Arquitectura Hexagonal para mantener nuestro dominio completamente agnóstico de la infraestructura.

Para el modelo de escritura (Command), mantuvimos una base de datos relacional altamente normalizada (RDS) para proteger las reglas de negocio y transacciones. Para el modelo de lectura (Query), creamos proyecciones desnormalizadas en una base de datos optimizada para lecturas rápidas (como DynamoDB).

El verdadero reto, como mencionas, fue la **consistencia eventual**. Lo manejé implementando un diseño orientado a eventos utilizando AWS (SNS y SQS). Cada vez que un comando modificaba el estado, el dominio emitía un evento. Un _worker_ asíncrono escuchaba este evento y actualizaba el modelo de lectura. Para mitigar la latencia de sincronización frente al usuario (esos milisegundos de desfase), trabajamos junto al equipo de Front-End (React/Next.js) para implementar actualizaciones optimistas en la UI. Además, a nivel de _backend_, implementé _Dead-Letter Queues_ (DLQ) y estrategias de reintentos automatizados para garantizar que, en caso de un fallo en la red, los modelos de lectura nunca quedaran permanentemente desincronizados."

**Resultado (Result):**
"La complejidad adicional valió cada segundo de esfuerzo. Al desacoplar la carga de la base de datos principal, logramos reducir la latencia de nuestras APIs de lectura en un 35%. Además, el uso de la Arquitectura Hexagonal permitió que los equipos de desarrollo pudieran escalar e iterar sobre las vistas de lectura sin tocar ni poner en riesgo la delicada lógica de negocio transaccional. Logramos un sistema sumamente rápido y resiliente con un _uptime_ del 99.9%."
